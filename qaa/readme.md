Q1. "랭체인 AgentExecutor를 쓰면 간편한데, 프로덕션 환경에서 발생할 수 있는 무한 루프(Infinite Loop)나 환각(Hallucination) 문제를 코드로 어떻게 완벽하게 통제할 건가요?"
•	❌ 탈락하는 답변: "어... 프롬프트에 '절대 무한 루프 돌지 말고 모르면 모른다고 해라'라고 적고, max_iterations=5 옵션을 주면 랭체인이 알아서 막아줍니다."
•	⭕ 합격하는 정석 답변:
"프롬프트나 프레임워크 옵션에만 의존하는 것은 프로덕션 환경에서 매우 위험합니다. 저는 이를 제어하기 위해 Raw API 레벨에서 상태 머신(State Machine) 아키텍처를 직접 구현하거나 LangGraph를 도입합니다.
에이전트에게 완벽한 자율 추론(ReAct) 권한을 주는 대신, **[동적 라우팅 노드]**를 촘촘히 설계하여 특정 단계에서는 오직 지정된 툴 그룹만 매싱되도록 컨텍스트를 강제 제한합니다. 또한 내부 실행 루프에 하드코딩된 예외 처리(Try-Catch) 레이어를 두어, 타임아웃이나 파싱 에러가 발생하면 LLM의 판단을 거치지 않고 시스템이 즉시 세이프넷(Fallback) 노드로 흐름을 꺾도록 설계합니다."
Q2. "도구(Tool)의 개수가 30개가 넘어가면 LLM이 파라미터를 잘못 매핑하거나 엉뚱한 툴을 호출하는 경향이 있습니다. 이 성능 저하를 시스템 아키텍처적으로 어떻게 해결하시겠습니까?"
•	❌ 탈락하는 답변: "도구 설명을 더 자세하게 한글/영어 주석으로 길게 적어주고 GPT-4o 같은 가장 좋은 모델을 씁니다."
•	⭕ 합격하는 정석 답변:
"LLM의 컨텍스트 윈도우에 한 번에 30개의 도구 명세(JSON Schema)를 다 밀어 넣으면 'Attention Distraction(주의력 분산)' 현상이 발생합니다. 저는 이를 '동적 도구 라우팅(Dynamic Tool Selection)' 패턴으로 해결합니다.
1차적으로 사용자의 의도(Intent)를 분류하는 가벼운 라우터(Router) 모델을 앞에 두거나 벡터 DB(RAG)를 활용해, 현재 맥락에 꼭 필요한 3~5개의 도구 명세만 실시간으로 필터링해서 LLM의 tools 파라미터에 탑재합니다. 이로써 LLM이 읽어야 할 설명서의 양을 최소화하여 인퍼런스 비용을 줄이고 툴 호출 정확도를 극적으로 끌어올립니다."
Q3. "에이전트의 프롬프트를 한 줄 수정했습니다. 이 수정이 기존 시스템의 다른 에이전트 워크플로우를 망가뜨리지 않았다는 것을 코드로 어떻게 검증(Test)할 건가요? 라인 바이 라인 디버깅이 안 되잖아요?"
•	❌ 탈락하는 답변: "질문을 여러 개 다시 던져보고 잘 돌아가는지 눈으로 테스트(Manual QA) 해봅니다."
•	⭕ 합격하는 정석 답변:
"에이전트는 비결정적이기 때문에 기존의 단위 테스트 방식은 통하지 않습니다. 저는 LangSmith/Langfuse의 SDK를 CI/CD 파프라인에 연동하여 해결합니다.
프롬프트가 변경되면 미리 구축해 둔 50~100개의 에이전트 시나리오 데이터셋(Golden Dataset)을 백그라운드에서 자동으로 전수 실행합니다. 이후 'LLM-as-a-Judge(평가자 모델)' 기법을 도입하여, 수정된 에이전트가 뱉은 툴 호출 순서와 최종 답변의 정확도를 평가지표(Accuracy, Hallucination Score)로 수치화합니다. 이 자동화된 회귀 테스트(Regression Test)에서 스코어가 95% 이상 유지될 때만 메인 브랜치에 코드를 머지합니다."
